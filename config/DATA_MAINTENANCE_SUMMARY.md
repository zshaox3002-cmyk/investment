## 一、问题诊断

### 1.1 当前数据结构的缺陷

**发现的问题**：
1. ❌ **holdings.csv** — 仅包含 C 档个股（7 只），缺少 A/B 档数据
2. ❌ **core_etf.csv** — 仅包含 B 档 ETF 配置，大部分为 0 份额
3. ❌ **cash_positions.csv** — 完全缺失，无 A 档现金/短债数据
4. ❌ **portfolio_snapshot.csv** — 不存在，无完整投资组合快照
5. ❌ **portfolio_history.csv** — 不存在，无历史追踪数据
6. ❌ **portfolio_transactions.csv** — 不存在，无交易日志审计

**影响**：
- 无法准确计算总组合的回撤和收益
- 无法验证 A/B/C 配置是否符合目标
- 无法追踪资金流向和交易历史
- C 仓位重组的分析基础不完整

### 1.2 为什么需要补充 A 档数据

**根据 CLAUDE.md 的资金结构**：
```
A 档（25%）：生活保障金 = 现金 + 货币基金 + 短债基金
B 档（50%）：核心 ETF = 沪深300 + 中证500 + 标普500 + 纳指100 + 黄金
C 档（25%）：主动选股 = 7 只个股
```

**当前实际情况**：
```

B 档（12.86%）：严重低配 ❌
C 档（87.14%）：严重超配 ❌
```

**问题的严重性**：
- 组合结构完全失衡
- 无应急现金储备
- 无防御性资产
- 风险极高

### 2.2 数据维护机制

**核心原则**：
- 🔄 **自动化优先** — 日度数据自动生成
- 📝 **半自动化** — 交易后自动记录，人工验证
- 👤 **人工审查** — 月度和季度进行完整验证

**更新频率**：

| 频率 | 触发条件 | 执行脚本 | 时间投入 |
|------|---------|---------|---------|
| **日度** | 每个交易日收盘后 | daily_snapshot.py | 5-10 分钟 |
| **交易后** | 每笔交易执行后 | trade_log.py | 5-10 分钟 |
| **月度** | 每月最后一个交易日 | monthly_update.py | 30-60 分钟 |
| **季度** | 每季度最后一个交易日 | quarterly_review.py | 4-6 小时 |

## 四、数据维护的关键要素

### 4.1 数据来源

**股票价格**：
- 来源：交易所 / 数据 API
- 更新频率：日度（收盘后）
- 获取方式：自动 API / 手动输入

**基金净值**：
- 来源：基金公司 / 数据 API
- 更新频率：日度（收盘后）
- 获取方式：自动 API / 手动输入

**现金余额**：
- 来源：银行 / 基金公司
- 更新频率：日度（手动查询）
- 获取方式：手动输入

### 4.2 数据验证

**日度验证公式**：
```
总市值 = A 档市值 + B 档市值 + C 档市值

A 档市值 = 现金 + 货币基金 + 短债基金
B 档市值 = Σ(ETF 份额 × 当日净值)
C 档市值 = Σ(个股股数 × 当日收盘价)

总仓位占比 = 100%
A 档占比 = A 档市值 / 总市值
B 档占比 = B 档市值 / 总市值
C 档占比 = C 档市值 / 总市值
```

**异常判断标准**：
- 总市值偏差 > 1% → 检查数据源
- 仓位占比偏差 > 2% → 检查是否有未记录交易
- 成本价异常 → 检查是否有计算错误

### 4.3 异常处理

**异常处理流程**：
```
发现异常
    ↓
确认异常类型
    ├─ 数据输入错误 → 修正输入
    ├─ 计算错误 → 修正公式
    ├─ 未记录交易 → 补充记录
    └─ 系统错误 → 调试脚本
    ↓
修正后重新运行
    ↓
验证异常是否消除
    ↓
记录异常和处理过程
```
